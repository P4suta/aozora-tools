## アーキテクチャ (refactor 後)

```
DocState
├── buffer: parking_lot::Mutex<BufferState>   ← writer-only (apply_changes / replace_text)
│   ├── text: String
│   ├── incremental: IncrementalDoc            ← TS parser + tree
│   └── segment_cache: SegmentCache            ← 意味論パース結果
├── snapshot: ArcSwap<Snapshot>                ← reader-only, wait-free
│   └── Snapshot
│       ├── text: Arc<str>
│       ├── line_index: Arc<LineIndex>
│       ├── gaiji_spans: Arc<BTreeMap<u32, GaijiSpan>>  ← BTree で incremental update 準備
│       └── version: u64
├── edit_version: AtomicU64                    ← write 時に bump
└── metrics: Arc<Metrics>
```

**Read path** (全 LSP request handler):
```rust
let state = self.lookup(&uri)?;  // DashMap shard read (μs) + Arc::clone
let snap = state.snapshot();     // ArcSwap::load_full → 8 ns wait-free
// snap.text, snap.line_index, snap.gaiji_spans を自由にアクセス
```

**Write path** (`did_change`):
```rust
state.apply_changes(&edits);  // 内部で:
  // 1. buffer mutex 取得 (μs)
  // 2. text mutate + TS apply_edit (200 ms; ★ 他 writer のみブロック)
  // 3. edit_version bump
  // 4. mutex drop
  // 5. tokio::spawn_blocking で snapshot rebuild (LineIndex + gaiji_spans)
  // 6. ArcSwap::rcu で atomic install (古い版なら自然に dropped)
```

Reads は writer の 200ms に**完全に独立**。

## 次セッションでまずやること

優先度 1: **#190 増分 gaiji rebuild** (algorithmic 最後の山場)

設計案:
```rust
pub struct DocState {
    ...
    pending_edits: Mutex<Vec<InputEdit>>,  // 新規
}

impl DocState {
    pub fn apply_changes(...) -> Option<u64> {
        // ... text mutate + TS apply_edit ...
        self.pending_edits.lock().push(ts_edit);  // 追加
        // ... bump version, spawn rebuild ...
    }

    pub fn rebuild_snapshot_now(&self) {
        let edits = std::mem::take(&mut *self.pending_edits.lock());
        let (text, new_tree, version) = self.snapshot_buffer_inputs();
        let old_snapshot = self.snapshot.load();
        let new_spans = if let Some(old_tree_arc) = old_snapshot.tree.clone() {
            extract_gaiji_spans_incremental(
                &old_tree_arc,
                &new_tree,
                &old_snapshot.gaiji_spans,
                &edits,
                &text,
            )
        } else {
            extract_gaiji_spans(&new_tree, &text)  // cold start
        };
        // ... build snapshot, rcu install ...
    }
}
```

`extract_gaiji_spans_incremental` algorithm:
1. `old_tree.changed_ranges(&new_tree)` で changed byte ranges (in new tree coords)
2. 各 old_span について:
   - edits を順次適用して new byte range を計算 (累積 byte shift)
   - new range が changed_ranges のいずれかと交差 → drop (再抽出)
   - 否則 → 新オフセットで carry forward
3. 各 changed_range で new_tree を sub-walk して新規 spans 抽出

期待される改善: typical edit (cursor in middle) で 67ms → ~5ms (重なる範囲のみ walk)。
worst case (offset 0 insert) は依然 67ms (全範囲 changed)。

優先度 2 以降: ユーザに「他にやりたいこと」を聞く。今回 ArcSwap refactor が完了したので、
LSP 機能拡張 (semantic tokens / folding range / document symbol) や docs ADR 化が候補。
