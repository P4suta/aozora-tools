## アーキテクチャ (refactor 後)

```
DocState
├── buffer: parking_lot::Mutex<BufferState>      ← writer-only
│   ├── text: String
│   ├── incremental: IncrementalDoc               ← TS parser + tree
│   └── segment_cache: SegmentCache               ← 意味論パース結果
├── snapshot: ArcSwap<Snapshot>                   ← reader-only, wait-free
│   └── Snapshot
│       ├── text: Arc<str>
│       ├── line_index: Arc<LineIndex>
│       ├── gaiji_spans: Arc<BTreeMap<u32, GaijiSpan>>  ← incremental update 対応済
│       ├── tree: Option<Tree>                            ← 次回 changed_ranges の base
│       └── version: u64
├── edit_version: AtomicU64                       ← write 時に bump
├── pending_edits: Mutex<Vec<(u64, InputEdit)>>   ← incremental rebuild 用 edit log
└── metrics: Arc<Metrics>
```

## 次セッションでまずやることの候補 (優先順位なし)

1. **`Arc<GaijiSpan>` boxing** — incremental rebuild の真の win を引き出す。
   現在 carry-forward は `GaijiSpan { description: String, mencode: Option<String> }` を
   clone (50k * String alloc = 数十 ms)。`Arc<GaijiSpan>` 化すれば pointer bump のみ。
   `Snapshot::gaiji_spans: Arc<BTreeMap<u32, Arc<GaijiSpan>>>` への変更。
   Inlay handler 等の consumer を `&Arc<GaijiSpan>` 経由のアクセスに更新。

2. **TS reparse コスト削減** — apply_changes wall の 200ms (= 75%) を占める tree-sitter
   incremental parse 自体を高速化したい。候補:
   - Rope (`ropey`) buffer + `parser.parse_with(callback, ...)` で String materialise を回避
   - Per-paragraph segmentation (1 paragraph = 1 mini tree-sitter parse)
   - 文書を chunk 分割して TS を並列に走らせる

3. **LSP 機能拡張** — wait-free reads 基盤が整ったので、追加機能の cost が低い:
   - `textDocument/foldingRange` — `［＃ここから〜終わり］` 折りたたみ
   - `textDocument/documentSymbol` — outline view
   - `textDocument/semanticTokens` — TextMate より精密な color
   - `textDocument/codeLens` — chapter ごとの 字数 / page count

4. **VS Code 拡張機能の磨き込み**:
   - HTML preview pane (handoff plan 既述、未着手)
   - TextMate grammar (LSP 起動前の syntax color)
   - Snippets

5. **Docs / ADR 整備**:
   - aozora-tools README から旧 \`afm/\` 参照を削除
   - ADR-0001: LSP feature roadmap
   - ADR-0002: position encoding (UTF-16 column ↔ byte)
   - ADR-0003: ArcSwap snapshot architecture

## 試して却下した alternatives (data 駆動の判断)

- **tree-sitter `Query` API** for gaiji extraction → 5× 遅 (71 → 330 ms)、却下
- **Rope-backed buffer** → string copy は 0.3 ms とノイズレベル、利益薄、棚上げ
- **`tokio::task::spawn_blocking` 内部での hot path 最適化** → bench 不要レベル、棚上げ
