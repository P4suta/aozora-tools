### 4. Bench data に基づいて refactor 着手

仮説: `gaiji_span_extract` (TS tree walk) が apply_changes の dominant cost。これが bench で確認できれば、refactor 方針:

#### Option A: Snapshot ArcSwap (#187)

```rust
pub struct DocState {
    text: parking_lot::RwLock<String>,
    incremental: IncrementalDoc,        // own Mutex
    snapshot: ArcSwap<DocSnapshot>,     // atomic-swap
    parse_version: AtomicU64,
    metrics: Arc<Metrics>,
}

pub struct DocSnapshot {
    text: Arc<str>,
    line_index: Arc<LineIndex>,
    gaiji_spans: Arc<[GaijiSpan]>,
    version: u64,
}
```

- 読み: `state.snapshot.load()` → wait-free
- 書き: text + TS edit (高速、buffer lock) → 別 task で snapshot 計算 → atomic swap

#### Option B: Lazy snapshot (read-on-demand)

snapshot を didChange で計算せず、最初の read で計算 (OnceCell パターン)。read 中の cancel に注意。

#### Option C: Incremental gaiji span update

TS の `Tree::changed_ranges(old, new)` を使って、変更されたノードだけ extract 再計算。
全文 walk を回避。

**Bench データを見て A / B / C を決める**。

## architecture audit 要点 (次セッション参照用)

```
## State graph (per Url, inside DashMap shard)
DocState:
  text:           String          mutable on edit
  segment_cache:  SegmentCache    holds Vec<Diagnostic>; replaced via set_diagnostics
  incremental:    IncrementalDoc  TS parser+Tree, own Mutex<Inner>
  line_index:     LineIndex       replaced wholesale on edit
  gaiji_spans:    Arc<[GaijiSpan]> replaced on edit
  parse_version:  AtomicU64
  metrics:        Arc<Metrics>    own atomic counters + histogram Mutex

## Lock graph
L1 DashMap shard RwLock
   - writers: did_open, did_change, did_close, debounced parse task
   - readers: every request handler (hover, codeAction, gaiji_spans, completion, …)
L2 IncrementalDoc.inner Mutex
   - apply_edit, parse_full, with_tree
L3 Metrics histogram Mutex (cheap)

## Suspected hot path
did_change → docs.get_mut (L1 write) → DocState::apply_changes:
  1. text mutate (μs)
  2. TS apply_edit (L2 acquire + release, μs)
  3. LineIndex::new(text)        — ~6ms on 6MB SIMD
  4. refresh_gaiji_spans         — TS Mutex + tree walk + per-node parse
                                   ~30-50ms on bouten.afm (24k tokens)
  5. parse_version++
Total L1 hold time: ~40-60ms per edit
With 10 edits/sec + 32 concurrent reads → max read latency ~ 50ms × N waiting
matches user-observed 1-2 second responses during burst typing.

## Refactor target
Split DocState into:
  BufferState (mutable, brief lock):
    text, incremental
  Snapshot (immutable Arc, atomic-swap):
    text Arc<str>, line_index Arc<LineIndex>, gaiji_spans Arc<[GaijiSpan]>, version
Reads: snapshot.load() — wait-free.
Writes: update buffer (μs) → compute Snapshot (off-lock, may take 50ms) → atomic swap.
Reads NEVER wait for writes; writes NEVER wait for reads.
```
