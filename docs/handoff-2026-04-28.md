## アーキテクチャ最終形

```
DocState
├── buffer: parking_lot::Mutex<BufferState>      ← writer-only
│   ├── text: ropey::Rope                         ← O(log n) splice + chunked TS input
│   ├── incremental: IncrementalDoc               ← parse_full_rope / apply_edit_rope
│   └── segment_cache: SegmentCache
├── snapshot: ArcSwap<Snapshot>                   ← reader-only, wait-free 8ns
│   └── Snapshot
│       ├── text: Arc<str>                        ← rope → Arc<str> materialise once/rebuild
│       ├── line_index: Arc<LineIndex>
│       ├── gaiji_spans: Arc<BTreeMap<u32, Arc<GaijiSpan>>>  ← 構造体ごと Arc 共有
│       │       └── GaijiSpan { description: Arc<str>, mencode: Option<Arc<str>>, ... }
│       ├── tree: Option<Tree>                    ← changed_ranges の base
│       └── version: u64
├── edit_version: AtomicU64
├── pending_edits: Mutex<Vec<(u64, InputEdit)>>   ← incremental gaiji の edit log
└── metrics: Arc<Metrics>
```

**一切の暗黙の文字列コピーがない**:
- buffer mutate: O(log n) Rope edit (no String copy)
- TS reparse: chunked input (no String materialisation)
- Snapshot read: ArcSwap load (1 atomic, no copy)
- Snapshot rebuild: Rope→Arc&lt;str&gt; materialise once
- Carry-forward gaiji span: Arc::clone (1 atomic)

## 残タスク (次セッションへの引き継ぎ)

このセッションで挙げた 3 つすべて完了したので、以下は **新提案**:

### 優先度 1: TS reparse 200ms の根本対策

Bench 上 dominant な 200ms は tree-sitter incremental reparse 自体 (offset 0 insert 最悪ケース)。
これを下げるには:

- **per-paragraph segmentation** — 文書を「\n\n」で paragraph に分割し、各 paragraph を独立 tree-sitter parse。Edit が 1 paragraph に収まれば再 parse は局所のみ
- **tree-sitter grammar の simplify** — 現状 `_element` の choice にいくつものパターンがある。実用上不要なルールを削れば parse table が縮む
- **measure_incremental の improve** — typical edit (cursor in middle of small paragraph) で本当に 200ms 出るか measure。最悪ケースだけ高い可能性

### 優先度 2: VS Code 拡張で新 LSP feature を活用

Server side は capabilities を出しているが、editor 側で見せるには:
- `package.json contributes.commands` に "Aozora: outline" 等を追加 (内部的に LSP request を打つ)
- semantic tokens の theme color を VS Code theme で設定 (default theme で目立つよう試す)
- folding は VS Code 自動表示なので追加作業不要

### 優先度 3: ADR 整備

このセッションで landed したアーキテクチャ決定を 3 本の ADR に:
- ADR-0001: ArcSwap snapshot pattern (wait-free reads)
- ADR-0002: Rope-backed buffer (O(log n) edit + TS chunked input)
- ADR-0003: incremental gaiji rebuild via Tree::changed_ranges + Arc&lt;GaijiSpan&gt;
